## 1. 背景与问题定义

### 1.1 背景

当前目标是重新开发一个面向多硬件平台的通用测试框架，用统一的配置、执行模型和结果模型承载研发测试、循环可靠性测试和生产测试。

### 1.2 核心问题

新平台必须重点解决以下问题：

1. 多板卡测试逻辑重复建设，复用性差。
2. 测试执行方式不统一，TE/EE 使用门槛高。
3. 平台能力与测试逻辑耦合，导致难以扩展新板型。
4. 状态、日志、报告、看板之间缺少统一数据契约。
5. 现有资产可参考，但不应继续作为新平台内核的实现基础。

### 1.3 设计目标

1. 建立配置驱动的通用测试平台内核。
2. 支持 Linux SoC、远程设备，并为 MCU/Zephyr 预留扩展位。
3. 通过 Functions、Cases、Fixtures 三层模型承载测试资产。
4. 通过统一事件与结果模型驱动日志、报告和 Dashboard。
5. 新增测试项或板型时，不需要修改执行内核。

### 1.4 非目标

以下内容不属于当前版本设计目标：

- 复杂 Web 平台或云端管理后台
- 完整 AI Agent / MCP 集成
- 历史数据库建设
- 所有旧测试逻辑的一次性迁移
- 所有平台能力的一次性抽象到位

## 3. 术语与核心概念

### 3.1 术语定义

- Function：最小原子测试单元，只验证一个能力点
- Case：围绕单一模块或主题组织的一组 Function 编排
- Fixture：围绕具体场景组织的一组 Case 编排
- Board Profile：板级能力与资源声明，不承载测试判定标准
- Capability：平台对外提供的标准能力，如 network、serial、gpio
- Execution Context：执行期上下文，包含解析后的配置、能力和运行状态
- Execution Event：执行过程中的标准事件

### 3.2 核心约束

1. Function 不直接读取配置文件。
2. Case 不直接承担底层平台探测职责。
3. Fixture 不直接编排 Function 细节。
4. Scheduler 是唯一调度核心。
5. Dashboard、报告、日志消费统一结果与事件模型，而不是直接耦合 Runner 内部实现。

## 4. 系统上下文与边界

### 4.1 外部参与者

- 用户：通过 CLI 触发执行、查看结果
- 开发板：被测对象
- 外部依赖：系统命令、Python 包、驱动、日志系统

### 4.2 系统边界

系统负责：

- 解析配置
- 组织执行
- 调用平台能力
- 记录状态与结果
- 生成报告和提供 Dashboard 数据

系统不负责：

- 驱动开发
- 板卡固件烧录流程
- 产线 MES 对接逻辑
- 云端数据平台建模

### 4.3 上下文图

```text
Configuration -> Execution Kernel -> Platform Capabilities
   |                    |                    |
   v                    v                    v
Result Store        Event Stream         Device/OS/Remote
   |
   v
Dashboard / Report / Logs
```

## 5. 架构总览

### 5.1 分层架构

```text
Interface Layer
  CLI entrypoints and command adapters

Configuration Layer
  config loader, schema validation, config resolver

Execution Layer
  scheduler, case runner, fixture runner, retry/timeout/loop policies

Domain Layer
  specs, execution context, events, results

Platform Layer
  adapters, capabilities, board profile mapping

Observability Layer
  result store, event store, report generator, dashboard provider
```

### 5.2 分层职责

| 层 | 职责 | 不负责 |
| --- | --- | --- |
| Interface | 接收用户输入，构造执行请求 | 业务调度细节 |
| Configuration | 解析配置并产出统一执行配置 | 实际执行 |
| Execution | 执行任务图和调度策略 | 平台细节实现 |
| Domain | 提供稳定对象模型与契约 | 外部 IO |
| Platform | 屏蔽平台差异并提供标准能力 | 场景编排 |
| Observability | 记录、聚合、输出执行数据 | 执行控制 |

### 5.3 架构决策

当前版本采用以下决策：

1. 进程内 Python 调用为主，不以 CLI 字符串拼接作为核心执行协议。
2. Scheduler 统一负责顺序、并发、重试、超时和中止策略。
3. 配置先解析后执行，不在执行过程中分散读取 JSON。
4. 平台能力以 capability 为中心，不以“万能 BaseAdapter”暴露过宽接口。
5. 事件流和结果快照并存，分别服务于可观测性与当前状态读取。

### 5.4 待确认决策

- 远程执行当前版本是否纳入内核还是通过适配层预留接口
- CLI Dashboard 当前版本是否落地为最小版本
- Function 的发现机制采用显式注册还是约定目录扫描

## 6. 核心领域模型

### 6.1 配置模型

配置模型用于描述“系统应该如何执行”，目标是把分散的静态配置与运行时覆盖收敛为单一、稳定、可校验的执行输入。

#### 6.1.1 GlobalConfig

GlobalConfig 提供产品级元数据和全局默认运行参数。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| product.sku | string | 是 | 产品型号标识 |
| product.stage | string | 是 | 当前阶段，如 EVT/DVT/PVT |
| product.board_profile | string | 否 | 默认板型 profile |
| runtime.default_timeout | int | 否 | 默认超时，单位秒 |
| runtime.default_retry | int | 否 | 默认重试次数 |
| runtime.default_retry_interval | int | 否 | 默认重试间隔，单位秒 |
| observability.report_enabled | bool | 否 | 默认是否生成报告 |
| observability.dashboard_enabled | bool | 否 | 默认是否启用 Dashboard |

约束：

1. GlobalConfig 只提供默认值，不承载具体测试函数参数。
2. GlobalConfig 不定义板级接口路径。
3. GlobalConfig 缺失时可退回内置默认值，但必须记录来源。

#### 6.1.2 BoardProfile

BoardProfile 用于描述板级资源、能力和接口候选，是平台能力绑定的输入，而不是测试判定标准的来源。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| profile_name | string | 是 | profile 唯一名称 |
| platform | string | 是 | 平台类型，如 linux、remote |
| supported_cases | list[string] | 否 | 允许执行的 case 白名单 |
| interfaces | map[string, list[string]] | 否 | 接口候选，如 eth、uart、i2c |
| capabilities | map[string, object] | 否 | 能力声明与补充参数 |
| tools_required | list[string] | 否 | 平台依赖命令 |
| metadata | object | 否 | 备注、板卡信息等 |

约束：

1. BoardProfile 不包含吞吐、时延等测试阈值。
2. interfaces 只声明候选，不代表已绑定成功。
3. supported_cases 是准入约束，不是场景编排定义。

#### 6.1.3 FunctionInvocationSpec

FunctionInvocationSpec 表示某个 Case 中一次具体的函数调用定义。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| name | string | 是 | Function 注册名 |
| enabled | bool | 否 | 是否启用，默认 true |
| params | map[string, any] | 否 | 调用参数 |
| expect | object | 否 | 本次调用的结构化判定规则 |
| timeout | int | 否 | 本次调用超时 |
| retry | int | 否 | 本次调用重试次数 |
| retry_interval | int | 否 | 本次调用重试间隔 |
| required_capabilities | list[string] | 否 | 显式声明所需 capability |
| tags | list[string] | 否 | 标签 |

约束：

1. params 可以使用模板引用，但当前推荐仅用于运行时元数据拼接；板卡专属 Case 中的真实接口值应直接写在 Case 资产里。
2. expect 用于定义该函数调用的通过/失败判定标准，必须使用统一规则结构，而不是任意自由格式。
3. Invocation 级 timeout/retry 可覆盖 Case 级默认值。
4. FunctionInvocationSpec 不定义平台实现细节。

##### 判定规则摘要

主设计文档只固定判定模型的边界和最小约束，不在此处展开完整 schema 细节。

expect 的最小语义要求如下：

1. expect 必须是结构化规则对象，不能是任意自由格式。
2. expect 至少需要表达规则聚合方式和规则列表两部分。
3. 每条规则至少需要表达待判定字段、判定操作和比较目标。
4. 待判定字段默认来自 function 返回结果中的 details 或 metrics。
5. 当前版本必须覆盖三类常见判定：bool 判定、数值阈值判定、数值区间判定。
6. 当前版本可以补充枚举判定和字段存在性判定。
7. 当前版本不支持在 expect 中写任意 Python 表达式或不可控脚本逻辑。

推荐的操作符族包括：等值比较、大小比较、区间比较、枚举比较、存在性判断。

详细字段定义、operator 清单、合法组合约束和完整 schema 示例，应下沉到后续的“配置 Schema 设计”子文档中维护。

#### 6.1.4 CaseSpec

CaseSpec 表示模块级测试编排单元。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| case_name | string | 是 | case 唯一名称 |
| module | string | 是 | 模块分类 |
| description | string | 否 | 描述 |
| functions | list[FunctionInvocationSpec] | 是 | 函数调用列表 |
| execution | string | 否 | sequential 或 parallel |
| timeout | int | 否 | case 总超时 |
| retry | int | 否 | case 级重试次数 |
| retry_interval | int | 否 | case 级重试间隔 |
| stop_on_failure | bool | 否 | 失败是否中止后续 function |
| required_interfaces | map[string, object] | 否 | 预检必须解析的接口需求 |
| precheck | bool | 否 | 是否执行预检，默认 true |

约束：

1. CaseSpec 负责模块编排，并可以显式声明该板卡实际要测试的接口值。
2. required_interfaces 是预检需求声明，不是 function 参数的隐式注入源。
3. execution 仅描述本 case 内部的函数执行策略。
4. 测试判定标准归属 Case 资产管理，并按 functions[].expect 粒度声明，而不是放入 board profile。

#### 6.1.5 FixtureSpec

FixtureSpec 表示场景级编排单元。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| fixture_name | string | 是 | fixture 唯一名称 |
| description | string | 否 | 场景说明 |
| cases | list[string] | 是 | case 文件引用或 case 名称 |
| execution | string | 否 | fixture 内 case 执行策略 |
| timeout | int | 否 | fixture 总超时 |
| retry | int | 否 | fixture 级重试次数 |
| retry_interval | int | 否 | fixture 级重试间隔 |
| stop_on_failure | bool | 否 | case 失败是否中止 fixture |
| loop | bool | 否 | 是否循环执行 |
| loop_count | int | 否 | 循环次数 |
| loop_interval | int | 否 | 循环间隔 |
| report_enabled | bool | 否 | 是否生成报告 |
| sn_required | bool | 否 | 是否要求序列号 |

约束：

1. FixtureSpec 只做场景组合，不直接引用 function。
2. loop=true 时，loop_count 必须大于 0。
3. sn_required=true 时，执行请求必须提供序列号或显式测试阶段标识。

#### 6.1.6 ResolvedExecutionConfig

ResolvedExecutionConfig 是执行前唯一可信配置对象，由解析器生成并作为执行内核输入。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| request | object | 是 | 原始执行请求摘要 |
| global_config | GlobalConfig | 是 | 全局配置快照 |
| board_profile | BoardProfile | 是 | 选定板型配置 |
| fixture | FixtureSpec | 否 | 解析后的 fixture |
| cases | list[CaseSpec] | 是 | 本次执行使用的 case 集合 |
| resolved_runtime | object | 是 | timeout、retry、loop 等最终运行参数 |
| resolved_interfaces | object | 否 | 预检与运行态视图中的接口解析结果，推荐使用 `bound/declared` 语义 |
| capability_requirements | list[string] | 否 | 汇总后的能力需求 |
| config_sources | object | 是 | 字段来源追踪 |

约束：

1. 执行阶段不得再回头读取原始 JSON 参与决策。
2. 所有覆盖关系必须在 ResolvedExecutionConfig 中可追溯。
3. ResolvedExecutionConfig 必须可序列化，便于调试和报告附带快照。

### 6.2 执行模型

执行模型用于描述“系统如何运行”，目标是把结构化配置转为任务图、运行上下文和统一结果。

#### 6.2.1 ExecutionRequest

ExecutionRequest 代表一次用户侧触发的执行请求。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| request_id | string | 是 | 本次请求唯一标识 |
| target_type | string | 是 | function、case、fixture |
| target_name | string | 是 | 目标名称或路径 |
| cli_overrides | map[string, any] | 否 | CLI 覆盖参数 |
| board_profile | string | 否 | 显式指定板型 |
| sn | string | 否 | 序列号 |
| operator | string | 否 | 操作人 |
| trigger_source | string | 否 | 触发来源，如 cli/manual |

#### 6.2.2 ExecutionPlan

ExecutionPlan 是 Scheduler 的直接输入，描述任务图结构而非原始配置文本。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| plan_id | string | 是 | 任务图标识 |
| root_task | ExecutionTask | 是 | 根任务 |
| tasks | list[ExecutionTask] | 是 | 扁平化任务集合 |
| execution_policy | object | 是 | 调度策略摘要 |
| resource_requirements | object | 否 | 资源需求摘要 |

设计约束：

1. ExecutionPlan 必须由 Runner 构建，不由 CLI 直接产生。
2. Scheduler 只消费 ExecutionPlan，不直接解析原始配置文件。

#### 6.2.3 ExecutionTask

ExecutionTask 是最小调度单元，可表示 fixture、case、function 三种层级。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| task_id | string | 是 | 任务唯一标识 |
| task_type | string | 是 | fixture、case、function |
| parent_task_id | string | 否 | 父任务 ID |
| name | string | 是 | 任务名称 |
| execution_mode | string | 是 | sequential 或 parallel |
| timeout | int | 否 | 任务超时 |
| retry_policy | object | 否 | 重试策略 |
| stop_on_failure | bool | 否 | 是否中止后续同级任务 |
| payload | object | 是 | 任务执行所需数据 |
| dependencies | list[string] | 否 | 依赖任务 ID |

设计约束：

1. task_type 决定 payload 结构。
2. function 任务是实际执行边界；fixture/case 任务是组织和聚合边界。
3. dependencies 仅用于显式顺序约束，不替代 execution_mode。

#### 6.2.4 ExecutionContext

ExecutionContext 是任务运行时共享上下文。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| request_id | string | 是 | 对应执行请求 |
| plan_id | string | 是 | 对应任务图 |
| resolved_config | ResolvedExecutionConfig | 是 | 已解析配置 |
| adapter_registry | object | 是 | 平台适配器入口 |
| capability_registry | object | 是 | 已装配 capability |
| runtime_state | object | 是 | 当前循环、重试、已完成任务等 |
| resource_locks | object | 否 | 当前占用资源状态 |
| artifacts_dir | object | 是 | logs/tmp/reports 目录上下文 |

设计约束：

1. ExecutionContext 是运行时对象，不作为长期配置存储。
2. 运行时状态与静态配置必须分离。
3. Context 中所有可变字段更新都应能触发事件或快照刷新。

#### 6.2.5 ExecutionResult

ExecutionResult 是统一返回结果模型，适用于 function、case、fixture 三个层级。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| task_id | string | 是 | 对应任务 ID |
| task_type | string | 是 | fixture、case、function |
| name | string | 是 | 任务名称 |
| status | string | 是 | passed、failed、timeout、skipped、aborted |
| code | int | 否 | 标准状态码 |
| message | string | 否 | 人类可读摘要 |
| details | map[string, any] | 否 | 详细信息 |
| metrics | map[string, number] | 否 | 测量结果 |
| artifacts | list[object] | 否 | 产物引用 |
| started_at | string | 是 | 开始时间 |
| finished_at | string | 是 | 结束时间 |
| duration_ms | int | 是 | 耗时 |
| retry_count | int | 否 | 实际重试次数 |
| children | list[ExecutionResult] | 否 | 子结果，主要用于 case/fixture |

设计约束：

1. status 用于调度与展示，code 用于兼容标准状态码语义。
2. case/fixture 结果由子结果聚合生成，但也必须有独立摘要字段。
3. ExecutionResult 必须可直接用于报告生成，不要求再二次解析日志。

### 6.3 可观测模型

可观测模型用于描述“系统发生了什么”和“当前状态是什么”。

#### 6.3.1 ExecutionEvent

ExecutionEvent 表示一次标准化生命周期事件，是事件流的基础对象。

#### 6.3.2 ResultSnapshot

ResultSnapshot 表示当前状态快照，供 Dashboard、CLI 和外部读取。

#### 6.3.3 EventRecord

EventRecord 表示已落盘事件记录，通常等于 ExecutionEvent 加上序列号、存储元信息。

#### 6.3.4 ReportArtifact

ReportArtifact 表示文本报告、JSON 报告或其他附属产物引用。

#### 6.3.5 DashboardSnapshot

DashboardSnapshot 表示经过聚合后的展示数据模型，而不是直接暴露底层结果对象。

### 6.4 模型关系

```text
FixtureSpec
  contains many CaseSpec
CaseSpec
  contains many FunctionInvocationSpec
ResolvedExecutionConfig
  combines global + board + fixture + case + cli override
ExecutionContext
  references resolved config + capabilities + runtime state
ExecutionResult
  produced by function/case/fixture execution
ExecutionEvent
  emitted during lifecycle transitions
```

## 7. 配置设计

### 7.1 配置来源

配置优先级必须明确且唯一：

1. global config
2. board profile
3. fixture config
4. case config
5. function defaults
6. CLI override

优先级规则：后者仅能覆盖前者允许覆盖的字段，不能改变对象类别和结构层级。

### 7.2 配置职责分离

- global config：全局默认值与产品级元数据
- board profile：板级资源、能力、接口候选、依赖声明
- fixture config：场景级调度策略和 case 组合
- case config：模块级 function 编排、模块级策略与测试判定标准
- function defaults：函数默认入参和能力声明
- CLI override：本次执行的临时覆盖

### 7.3 配置合并规则

#### 7.3.1 标量字段

标量字段按优先级链逐层覆盖，例如 timeout、retry、report_enabled。

#### 7.3.2 对象字段

对象字段采用浅层合并加白名单覆盖策略：

1. 已知对象结构由 schema 固定。
2. 仅允许覆盖声明为 overrideable 的子字段。
3. 未知字段或类型变化视为配置错误。

#### 7.3.3 列表字段

列表字段默认不拼接，按语义分别处理：

- cases：由 fixture 明确给出，不从 global 继承
- functions：由 case 明确给出，不从 board profile 继承
- tools_required：可由 global 和 board profile 去重合并

#### 7.3.4 模板解析

配置解析完成后允许对以下占位符做一次模板展开：

- ${product.sku}
- ${request.sn}
- ${resolved.interfaces.<type>.<name>}
- ${runtime.<field>}

模板解析原则：

1. 只允许引用已解析上下文。
2. 不支持任意表达式执行。
3. 对于当前 board-local Case，真实接口参数应优先直接写入 Case；`resolved.interfaces.*` 主要用于预检视图、运行态追踪或少量元数据拼接，推荐消费 `bound/declared` 字段。
4. 解析失败时直接报配置错误，不进入执行阶段。

### 7.4 配置原则

1. 测试判定标准不放在 board profile。
2. board profile 负责声明板级资源清单；Case 可以显式指定本次实际要测的接口值。
3. 测试判定标准默认放在 case.functions[].expect，而不是放在 function 实现代码中硬编码。
4. CLI 只能覆盖允许覆盖的运行参数，不能破坏配置结构。
5. 所有配置必须先通过 schema 校验。
6. 配置解析必须输出来源追踪信息。

### 7.5 配置文件定义

- config/global_config.json
- config/boards/<profile>.json
- cases/*.json
- fixtures/*.json

#### 7.5.1 GlobalConfig 示例

## 8. 执行内核设计

### 8.1 执行职责拆分

- FixtureRunner：将 FixtureSpec 转换为场景任务图
- CaseRunner：将 CaseSpec 转换为模块任务图
- Scheduler：统一调度和生命周期管理
- FunctionExecutor：执行单个 Function 并产出标准结果

职责边界：

1. Runner 负责构建任务图，不负责底层并发调度。
2. Scheduler 负责执行，不负责配置解析。
3. FunctionExecutor 负责单个函数调用，不负责跨函数编排。

### 8.2 调度能力

Scheduler 当前版本必须支持：

- sequential
- parallel
- timeout
- retry
- loop
- stop_on_failure

当前版本不支持：

- 分布式调度
- 跨主机任务抢占
- 优先级抢占式执行

### 8.3 生命周期

```text
request_received
  -> config_resolved
  -> fixture_started
  -> case_started
  -> function_started
  -> function_finished
  -> case_finished
  -> fixture_finished
```

生命周期要求：

1. 每个任务的 started 和 finished 必须成对出现，除非任务在创建前失败。
2. retry 不创建新的 request_id，但会创建新的 attempt 序号。
3. aborted 会终止未开始的同级或下游任务，并为其产出 skipped 或 aborted 结果。

### 8.4 执行策略

#### 8.4.1 sequential

同级任务按声明顺序执行。前一个任务结束后，后一个任务才可开始。

#### 8.4.2 parallel

同级任务并发执行，但必须受资源约束和 stop_on_failure 传播规则控制。

#### 8.4.3 retry

重试策略以任务级别定义，支持以下字段：

- max_attempts
- interval_seconds
- retry_on_status

默认规则：

1. passed 不重试。
2. config_error 不重试。
3. timeout、device_error、transient_failure 可重试。

#### 8.4.4 timeout

超时分两层：

- function_timeout：单次函数执行边界
- case_timeout / fixture_timeout：聚合边界

聚合超时触发时，调度器应主动中止仍在运行的子任务，并写入 aborted 事件。

#### 8.4.5 stop_on_failure

传播规则：

1. function 失败且 case.stop_on_failure=true，则同 case 下未开始任务标记 skipped。
2. case 失败且 fixture.stop_on_failure=true，则同 fixture 下未开始 case 标记 skipped。
3. parallel 模式下，已开始的同级任务不强制硬杀，由调度器按取消策略处理。

### 8.5 任务图构建

#### 8.5.1 FixtureRunner

输出 fixture 根任务及 case 子任务集合。

#### 8.5.2 CaseRunner

输出 case 任务及 function 子任务集合，并在进入调度前完成：

- 预检
- 接口绑定
- 模板替换
- capability 可用性确认

#### 8.5.3 FunctionExecutor

FunctionExecutor 输入为单个 function 任务和运行上下文，输出为 ExecutionResult。

执行边界：

1. 负责参数准备与调用。
2. 负责捕获异常、超时与返回值转换。
3. 负责生成 function 级结果和事件。

### 8.6 并发与取消模型

当前版本建议采用受控线程池或任务执行器模型，原因如下：

1. function 执行以 IO 和外部命令为主。
2. 线程池足以承载当前版本的并发复杂度。
3. 相比多进程，线程池更易共享 ExecutionContext 和事件总线。

取消规则：

1. 取消请求优先标记任务状态，再向下游传播。
2. 已启动外部命令的取消必须有超时回收兜底。
3. 取消是显式状态，不与失败混用。

### 8.7 结果聚合

#### 8.7.1 function -> case

Case 结果由其子 function 结果按以下规则聚合：

- 任一子任务 timeout，case 至少为 timeout 或 failed
- 任一子任务 failed 且 stop_on_failure 触发，case 为 failed
- 全部 passed，case 为 passed
- 部分 skipped，case 可为 passed_with_skip 或 failed，具体由策略确定

#### 8.7.2 case -> fixture

Fixture 结果按 case 聚合，规则与 case 聚合类似，但需附加：

- loop 执行时汇总每轮结果
- 生产模式下 pass/fail 口径优先简化，不暴露过多中间状态给用户

### 8.8 错误处理原则

1. 配置错误在执行前失败。
2. Function 失败由调度策略决定是否重试或中止。
3. 超时、依赖缺失、资源冲突必须是可区分的错误类型。
4. 失败必须保留最小可诊断上下文。

补充要求：

5. 执行异常必须映射到统一错误分类，而不是直接暴露原始异常给上层。
6. 重试前必须发出 retrying 事件，并记录 attempt 编号。
7. 聚合层失败不能吞掉下层原始失败信息。

### 8.9 待补充内容

- 资源锁是否在当前版本中强制实现为独立模块
- 外部命令取消与子进程回收细节
- passed_with_skip 是否进入当前版本状态枚举

## 10. Function / Case / Fixture 契约

### 10.1 Function 契约

每个 Function 至少应声明：

- name
- description
- input schema
- required capabilities
- runtime options
- output schema

说明：

- Function 定义“如何测”和“返回什么”。
- Function 不定义具体项目或板型的通过阈值；具体判定标准由 Case 在调用时提供。
- Function 返回结果时，应尽量把可判定字段结构化输出到 details 或 metrics，供 expect 规则消费。

### 10.2 Case 契约

每个 Case 至少应声明：

- case_name
- module
- functions
- execution policy
- timeout / retry policy
- required resources

说明：

- Case 定义“测哪些函数、以什么顺序测、按什么标准判定通过”。
- 如果一个 Case 中多个 Function 的标准不同，应分别写在各自 functions[].expect 中。
- Case 中的 expect 应采用统一 operator 模型，避免不同模块定义不同判定语法。

### 10.3 Fixture 契约

每个 Fixture 至少应声明：

- fixture_name
- cases
- execution policy
- stop_on_failure
- loop policy
- report policy

### 10.4 返回结果约束

统一结果结构建议至少包含：

- code
- message
- details
- metrics
- artifacts
- duration

## 11. 状态、事件与结果模型

### 11.1 为什么需要事件模型

仅靠 tmp 当前快照无法支撑完整的调试、报告生成和细粒度看板，因此必须同时保留事件流。

事件模型承担以下职责：

1. 记录生命周期变化。
2. 承载重试、取消、超时等动态行为。
3. 为报告与 Dashboard 提供统一事实来源。
4. 为故障定位提供最小可追溯链路。

### 11.2 标准事件

- fixture_started
- fixture_finished
- case_started
- case_finished
- function_started
- function_retrying
- function_passed
- function_failed
- function_timeout
- execution_aborted

建议补充：

- config_resolved
- precheck_started
- precheck_failed
- resource_acquired
- resource_released

### 11.3 快照与事件的分工

- 快照：提供当前状态读取
- 事件：记录发生过什么和发生顺序

补充原则：

1. 快照可以被覆盖写，事件不能覆盖写。
2. 快照面向“现在是什么状态”，事件面向“发生了什么变化”。
3. 报告优先基于结果和事件生成，而不是基于快照倒推。

### 11.4 ExecutionEvent 字段定义

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| event_id | string | 是 | 事件唯一标识 |
| event_type | string | 是 | 事件类型 |
| request_id | string | 是 | 对应请求 ID |
| plan_id | string | 否 | 对应任务图 ID |
| task_id | string | 否 | 对应任务 ID |
| task_type | string | 否 | fixture、case、function |
| task_name | string | 否 | 任务名称 |
| parent_task_id | string | 否 | 父任务 ID |
| attempt | int | 否 | 当前尝试次数，从 1 开始 |
| status_before | string | 否 | 变更前状态 |
| status_after | string | 否 | 变更后状态 |
| timestamp | string | 是 | 事件发生时间 |
| message | string | 否 | 摘要信息 |
| payload | object | 否 | 附加上下文 |

字段约束：

1. event_id 必须全局唯一。
2. timestamp 必须使用统一时间格式，建议 ISO 8601 UTC。
3. payload 必须是结构化对象，不允许写入未经控制的大段文本日志。

### 11.5 EventRecord 存储要求

EventRecord 在 ExecutionEvent 基础上增加持久化元数据：

- sequence：单次 request 内递增序号
- stored_at：落盘时间
- source：事件来源模块

存储约束：

1. sequence 必须单调递增。
2. 同一 request 内允许按 sequence 重建时间线。
3. 事件写入失败不能影响主执行流程，但必须记录告警日志。

### 11.6 状态机定义

#### 11.6.1 任务状态枚举

- pending
- running
- retrying
- passed
- failed
- timeout
- skipped
- aborted

#### 11.6.2 合法状态转换

| 当前状态 | 可迁移到 |
| --- | --- |
| pending | running, skipped, aborted |
| running | passed, failed, timeout, retrying, aborted |
| retrying | running, failed, timeout, aborted |
| passed | 终态 |
| failed | 终态 |
| timeout | 终态 |
| skipped | 终态 |
| aborted | 终态 |

约束：

1. 终态不可再次迁移。
2. retrying 不是独立结果，而是中间状态。
3. skipped 只用于未开始任务。

### 11.7 事件载荷约定

#### 11.7.1 config_resolved

payload 建议包含：

- target_type
- target_name
- selected_board_profile
- resolved_case_count
- config_hash

#### 11.7.2 function_started

payload 建议包含：

- required_capabilities
- resolved_interfaces
- timeout
- attempt

#### 11.7.3 function_failed / function_timeout

payload 建议包含：

- code
- error_type
- error_message
- retryable
- artifact_refs

#### 11.7.4 function_retrying

payload 建议包含：

- current_attempt
- next_attempt
- retry_interval_seconds
- last_error_type

### 11.8 ResultSnapshot 结构

ResultSnapshot 至少包含：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| request_id | string | 执行请求 ID |
| current_status | string | 当前整体状态 |
| fixture | object | fixture 级当前摘要 |
| cases | list[object] | case 级当前摘要 |
| updated_at | string | 最近更新时间 |
| counters | object | passed/failed/running/skipped 数量 |

快照要求：

1. 快照写入采用原子替换。
2. 快照必须可由事件和结果重新构建。
3. 快照字段应保持稳定，避免频繁影响 Dashboard。

### 11.9 结果聚合与展示口径

#### 11.9.1 内部口径

内部口径保留全部细粒度状态，包括 retrying、aborted、skipped。

#### 11.9.2 外部展示口径

面向普通 CLI 输出和生产场景时，可折叠为：

- PASS
- FAIL
- RUNNING
- SKIPPED

#### 11.9.3 报告口径

报告中必须同时保留：

- 聚合结果
- 子任务失败明细
- 重试次数
- 关键指标和产物引用

- DashboardSnapshot 字段最小集合
- 跨进程事件写入的一致性方案

## 14. 目录结构建议

```text
framework/
├── cli/
├── config/
├── domain/
├── execution/
├── platform/
│   ├── adapters/
│   └── capabilities/
└── observability/

functions/
cases/
fixtures/
config/
tests/
tools/
scripts/
doc/
```

### 14.1 目录原则

- 目录按职责划分，不按历史实现堆叠
- 如需迁移旧资产，迁移脚本应放在独立工具目录，而不是进入 framework 主路径
- tests 与 framework 结构尽量对应
- framework 只放平台内核代码，不放业务化脚本和一次性迁移逻辑

### 14.2 framework 目录细化

#### 14.2.1 cli

职责：

- 接收命令行输入
- 解析参数并构造 ExecutionRequest
- 调用应用层或执行入口

建议首批文件：

```text
framework/cli/
├── __init__.py
├── run_fixture.py
├── run_case.py
├── run_function.py
├── dashboard.py
└── common.py
```

文件说明：

- run_fixture.py：fixture 主入口
- run_case.py：case 主入口
- run_function.py：function 主入口
- dashboard.py：dashboard 入口
- common.py：共享参数、错误输出、请求构造

#### 14.2.2 config

职责：

- 加载配置文件
- 进行 schema 校验
- 合并配置并生成 ResolvedExecutionConfig
- 解析模板和来源追踪

建议首批文件：

```text
framework/config/
├── __init__.py
├── loader.py
├── models.py
├── resolver.py
├── validator.py
└── errors.py
```

文件说明：

- loader.py：负责文件读取与路径解析
- models.py：定义配置模型
- resolver.py：实现优先级合并、模板展开、接口绑定输入准备
- validator.py：负责 schema 校验
- errors.py：定义配置阶段错误类型

#### 14.2.3 domain

职责：

- 承载稳定领域对象
- 定义执行请求、任务、结果、事件等契约
- 避免业务逻辑散落到多处

建议首批文件：

```text
framework/domain/
├── __init__.py
├── requests.py
├── specs.py
├── execution.py
├── results.py
└── events.py
```

文件说明：

- requests.py：ExecutionRequest 等请求对象
- specs.py：FunctionInvocationSpec、CaseSpec、FixtureSpec
- execution.py：ExecutionPlan、ExecutionTask、ExecutionContext
- results.py：ExecutionResult、ResultSnapshot、ReportArtifact
- events.py：ExecutionEvent、EventRecord、状态枚举

#### 14.2.4 execution

职责：

- 构建任务图
- 统一调度任务执行
- 承载重试、超时、取消与聚合逻辑

建议首批文件：

```text
framework/execution/
├── __init__.py
├── scheduler.py
├── fixture_runner.py
├── case_runner.py
├── function_executor.py
├── policies.py
└── errors.py
```

文件说明：

- scheduler.py：唯一调度核心
- fixture_runner.py：FixtureSpec -> ExecutionPlan 根任务
- case_runner.py：CaseSpec -> function 子任务与预检
- function_executor.py：单个 function 执行边界
- policies.py：retry、timeout、stop_on_failure 等策略对象
- errors.py：执行阶段错误定义

#### 14.2.5 platform

职责：

- 识别平台与环境
- 注册平台能力
- 绑定 board profile 与 capability 实现

建议首批目录和文件：

```text
framework/platform/
├── __init__.py
├── registry.py
├── adapters/
│   ├── __init__.py
│   ├── base.py
│   ├── linux.py
│   └── remote.py
└── capabilities/
  ├── __init__.py
  ├── network.py
  ├── serial.py
  ├── gpio.py
  ├── i2c.py
  ├── rtc.py
  ├── storage.py
  ├── display.py
  └── system_info.py
```

文件说明：

- registry.py：adapter/capability 注册与查询
- adapters/base.py：适配器最小抽象
- adapters/linux.py：Linux 本地平台实现
- adapters/remote.py：远程执行预留实现
- capabilities/*.py：各能力最小统一接口与 Linux 首套实现

#### 14.2.6 observability

职责：

- 记录执行快照与事件
- 生成报告
- 为 Dashboard 提供聚合数据

建议首批文件：

```text
framework/observability/
├── __init__.py
├── result_store.py
├── event_store.py
├── report_generator.py
├── dashboard_provider.py
└── logger.py
```

文件说明：

- result_store.py：当前执行快照存取
- event_store.py：事件落盘与读取
- report_generator.py：从结果与事件生成报告

## 15. 非功能需求

### 15.1 可维护性

- 模块边界清晰
- 可替换、可测试
- 配置与执行分离

### 15.2 可靠性

- 关键结果原子落盘
- 失败可诊断
- 重试行为可追踪

### 15.3 性能

- 调度与监控不能明显干扰测试本身
- Dashboard 读取应尽量低侵入

### 15.4 可扩展性

- 新增板型不修改执行内核
- 新增 function 不修改调度器

### 15.5 安全性

- 外部命令执行需控制输入来源
- 配置与报告路径需避免路径注入
- 敏感凭据不写入日志

## 16. 风险与设计取舍

### 16.1 已知风险

1. 平台抽象过度，导致实现复杂度过高
2. 配置层设计不足，导致执行期继续散落取值
3. 事件模型缺失，导致 Dashboard 与报告再次耦合执行细节
4. 迁移期工具侵入新内核

### 16.2 应对策略

1. 先定义最小 capability 集合
2. 先完成配置解析层和执行内核，再扩展外围能力
3. 如需迁移旧资产，迁移逻辑放在 tools 或 scripts 下，避免侵入新内核
4. 用高频模块验证模型是否成立

## 18. 验收标准

1. 配置优先级明确且有测试覆盖。
2. Scheduler 成为唯一调度核心。
3. Function 不直接读取 board profile 或配置文件。
4. 平台差异集中在 platform/capabilities 层。
5. Dashboard 与报告仅依赖统一结果模型。
6. 新增常规测试模块无需修改核心执行链路。

- 报告格式是当前版本文本优先还是 JSON 优先
