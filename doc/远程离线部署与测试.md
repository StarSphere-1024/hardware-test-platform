# 远程离线部署与测试

本文档给新平台补齐一套可直接使用的远程闭环：本地打包源码与离线依赖、部署到开发板、执行单 case 或整套 fixture、最后回传 reports、logs 与 tmp。

## 设计取舍

当前仓库还没有根级 setup.py 或 pyproject.toml，也没有 console_scripts，因此这里采用更贴合现状的实现：

- 代码分发使用源码归档，不要求先做 Python 包发布。
- 远程执行直接调用 python -m framework.cli.run_case 与 python -m framework.cli.run_fixture。
- 依赖安装使用本地 wheelhouse 离线安装，开发板不访问外网。
- 部署时会排除本地已有的 wheelhouse/wheels 目录，避免远端源码树和依赖缓存各出现一份同名目录。
- psutil 默认尝试安装，用于提供更丰富的远程 dashboard 系统监控指标。如果远程板上没有 gcc、Python 头文件或其他构建条件，安装会失败并给出警告，但测试执行与 dashboard 主流程仍可运行，只是系统监控指标为空。

## 远程目录布局

部署脚本会在远程创建以下结构：

```text
REMOTE_DIR/
  framework/    # 解压后的源码目录之一
  functions/
  cases/
  fixtures/
  scripts/
  venv/         # 远程 Python 虚拟环境
  wheelhouse/   # 离线依赖包缓存
  reports/      # 远程执行产物
  logs/
  tmp/
```

CLI 在远程执行时统一使用：

```bash
python -m framework.cli.run_case
python -m framework.cli.run_fixture
```

这样不依赖打包产物，也不会和当前本地开发链路分叉。

## 参数传递约定

首次使用时，推荐优先使用命令行参数把远程目标写全；确认目标固定后，再通过环境变量省略重复输入。

1. 命令行参数方式

`scripts/run_remote_case.sh`、`scripts/run_remote_fixture.sh`、`scripts/fetch_remote_artifacts.sh`、`scripts/check_remote_status.sh` 支持命名参数：

```bash
scripts/run_remote_fixture.sh --host 192.168.100.119 --user seeed --password seeed --remote-dir /home/seeed/hardware_test_platform rk3576_smoke
```

`scripts/package_and_deploy_offline.py` 当前支持位置参数：

```bash
python scripts/package_and_deploy_offline.py 192.168.100.119 seeed seeed /home/seeed/hardware_test_platform --download-missing
```

2. 环境变量方式

适用于所有远程相关脚本：

```bash
export REMOTE_HOST=192.168.100.119
export REMOTE_USER=seeed
export REMOTE_PASS=seeed
export REMOTE_DIR=/home/seeed/hardware_test_platform
```

推荐约定：

- 临时执行或单次验证时，优先使用命令行参数，避免依赖当前 shell 环境。
- 连续回归或批量调试时，优先先导出环境变量，再运行各脚本。
- 如果同时提供了环境变量和命令行参数，脚本会以命令行参数为准。

## 一次完整闭环

### 1. 部署源码与离线依赖

首次部署：

```bash
python scripts/package_and_deploy_offline.py 192.168.100.119 seeed seeed /home/seeed/hardware_test_platform --download-missing
```

如果已经在当前 shell 中导出了 `REMOTE_HOST / REMOTE_USER / REMOTE_PASS / REMOTE_DIR`，也可以省略这些参数：

```bash
python scripts/package_and_deploy_offline.py --download-missing
```

如果希望远程 dashboard 尝试展示 CPU/内存等监控指标（默认行为），并且远程板具备 gcc 与 Python 编译环境：

```bash
python scripts/package_and_deploy_offline.py 192.168.100.119 seeed seeed /home/seeed/hardware_test_platform --download-missing
```

如果明确不需要 psutil 系统监控指标，可以使用 --skip-psutil 跳过：

```bash
python scripts/package_and_deploy_offline.py 192.168.100.119 seeed seeed /home/seeed/hardware_test_platform --download-missing --skip-psutil
```

快速复测，只同步源码：

```bash
python scripts/package_and_deploy_offline.py 192.168.100.119 seeed seeed /home/seeed/hardware_test_platform --fast-reuse
```

说明：

- --download-missing 会在本地 wheelhouse 不完整时，使用当前开发机的 pip 下载 rich 与 pyserial（以及 psutil 相关依赖，除非使用 --skip-psutil）。
- --skip-psutil 会跳过 psutil 的准备和安装，远程 dashboard 系统监控指标将不可用。
- 如果本地已经维护好了 wheelhouse，可以不传 --download-missing。
- --fast-reuse 会复用远端已有 venv 和依赖层；如果远端虚拟环境不存在，脚本会直接失败，并提示改用完整部署。
- 当前源码会直接解压到 REMOTE_DIR 根目录，不再额外包一层 workspace。
- 即使可选的 psutil 安装失败，主体部署仍会完成，但会明确输出 WARN，提示 dashboard 系统监控指标不可用。

### 2. 执行单个 case

```bash
scripts/run_remote_case.sh --host 192.168.100.119 --user seeed --password seeed --remote-dir /home/seeed/hardware_test_platform cases/rk3576/uart_case.json
```

如果已经导出了远程环境变量，也可以简写为：

```bash
scripts/run_remote_case.sh gpio_case
scripts/run_remote_case.sh cases/rk3576/eth_case.json
```

### 3. 执行整套 RK3576 smoke fixture

```bash
scripts/run_remote_fixture.sh --host 192.168.100.119 --user seeed --password seeed --remote-dir /home/seeed/hardware_test_platform rk3576_smoke
```

如果已经导出了远程环境变量，也可以简写为：

```bash
scripts/run_remote_fixture.sh rk3576_smoke
```

默认使用 [../fixtures/rk3576_smoke.json](../fixtures/rk3576_smoke.json)，它当前包含 RK3576 的 ETH、UART、RTC、I2C、GPIO 五个 board-local case。

RK3576 的 ETH 与 UART 已单独放在 [../cases/rk3576](../cases/rk3576) 下，但它们依赖额外现场条件：

- eth_case 在未插网线或对端不可达时会直接失败并上报，这属于有效测试结果，不做静默跳过。
- uart_case 在未接 loopback 工装或跳线时会直接失败并上报，这同样属于有效测试结果。

### 4. 回传产物

```bash
scripts/fetch_remote_artifacts.sh --host 192.168.100.119 --user seeed --password seeed --remote-dir /home/seeed/hardware_test_platform --output-dir remote-artifacts/rk3576_smoke_01
```

如果已经导出了远程环境变量，也可以简写为：

```bash
scripts/fetch_remote_artifacts.sh
scripts/fetch_remote_artifacts.sh remote-artifacts/rk3576_smoke_01
```

### 5. 检查远程状态

```bash
scripts/check_remote_status.sh --host 192.168.100.119 --user seeed --password seeed --remote-dir /home/seeed/hardware_test_platform
```

如果已经导出了远程环境变量，也可以简写为：

```bash
scripts/check_remote_status.sh
```

## 当前闭环能力

当前远程辅助脚本已经覆盖以下最小闭环能力：

- deploy: [../scripts/package_and_deploy_offline.py](../scripts/package_and_deploy_offline.py)
- run case: [../scripts/run_remote_case.sh](../scripts/run_remote_case.sh)
- run fixture: [../scripts/run_remote_fixture.sh](../scripts/run_remote_fixture.sh)
- fetch artifacts: [../scripts/fetch_remote_artifacts.sh](../scripts/fetch_remote_artifacts.sh)
- check status: [../scripts/check_remote_status.sh](../scripts/check_remote_status.sh)

当前实现的特点有两点：

- 当前方案暂时不依赖 wheel 安装业务代码，而是直接同步源码到 REMOTE_DIR 根目录。
- 当前方案直接跑源码 CLI 模块，不依赖 run_case、run_fixture console script。

## 后续建议

如果要进一步做到真正的一键远程回归，建议继续补两项：

1. 根级 pyproject.toml 与 console_scripts，把源码运行切换为标准安装模式。
2. 在 scripts/ 之上再封一层 orchestration 脚本，例如 remote_regression.sh，串起部署、fixture 执行、产物回传与结果判定。
