# 远程离线部署与测试指南

本文档旨在为新平台提供一套完整的**远程闭环测试方案**：实现本地源码打包、离线依赖同步、远程执行测试、以及测试产物（日志/报告）回传的自动化流程。

## 1. 核心设计思路
由于当前框架尚未进行标准 Python 包化（无 `setup.py`），我们采用了更灵活的“源码+离线依赖”方案：
- **分发方式**：直接同步源码归档，通过 `python -m` 调用模块。
- **离线支持**：利用本地 `wheelhouse` 缓存，实现远程板端无网安装。
- **性能监控**：可选安装 `psutil` 以支持 Dashboard 系统指标监控（依赖板端编译环境）。

---

## 2. 环境配置
为了简化命令输入，建议优先配置环境变量。脚本遵循：**命令行参数 > 环境变量 > 默认值**。

### 设置环境变量 (推荐)
```bash
export REMOTE_HOST=192.168.100.119
export REMOTE_USER=seeed
export REMOTE_PASS=seeed
export REMOTE_DIR=/home/seeed/hardware_test_platform
```

---

## 3. 标准操作闭环 (Step-by-Step)

### 第一步：部署源码与依赖
根据远程环境状态选择执行方式：

*   **首次部署（完整版）**：同步源码、下载并安装依赖。
    ```bash
    # --download-missing: 若本地无 wheel 缓存则在线下载
    python scripts/package_and_deploy_offline.py --download-missing
    # 或显式指定远程参数（覆盖环境变量）
    python scripts/package_and_deploy_offline.py 192.168.100.119 seeed seeed /home/seeed/hardware_test_platform --download-missing
    ```
*   **快速更新（源码版）**：仅同步源码，复用远程已有的虚拟环境。
    ```bash
    python scripts/package_and_deploy_offline.py --fast-reuse

    # 或显式指定远程参数（覆盖环境变量）
    python scripts/package_and_deploy_offline.py 192.168.100.119 seeed seeed /home/seeed/hardware_test_platform --fast-reuse
    ```
*   **精简部署**：若远程缺少编译环境，跳过 `psutil` 安装。
    ```bash
    python scripts/package_and_deploy_offline.py --skip-psutil
    ```

### 第二步：执行测试
支持运行单个 Case 或整套 Fixture。

*   **运行单个 Case**：
    ```bash
    scripts/run_remote_case.sh cases/rk3576/uart_case.json
    ```
*   **运行整套 Fixture**：
    ```bash
    scripts/run_remote_fixture.sh rk3576_smoke
    ```

补充说明：上述远程脚本会先 `cd` 到 `REMOTE_DIR`，再调用 `python -m framework.cli.*`。当前 CLI 已支持自动探测项目根，因此脚本内部不再需要显式传入 `--workspace-root` 或 `--artifacts-root`。

### 第三步：回传测试产物
测试完成后，将远程生成的 `reports/`、`logs/` 和 `tmp/` 目录同步至本地。
```bash
# 默认回传至本地 remote-artifacts/ 目录
scripts/fetch_remote_artifacts.sh
# 或指定本地存储子目录
scripts/fetch_remote_artifacts.sh my_test_results
```

### 第四步：状态检查 (可选)
检查远程目录结构及进程状态。
```bash
scripts/check_remote_status.sh
```

## 4. 远程目录布局结构
部署完成后，远程目标机（`REMOTE_DIR`）的目录结构如下：

```text
REMOTE_DIR/
├── framework/      # 测试框架核心代码
├── functions/      # 业务功能封装
├── cases/          # 测试用例定义 (JSON)
├── fixtures/       # 测试集定义 (JSON)
├── venv/           # 远程 Python 虚拟环境
├── wheelhouse/     # 离线依赖包 (.whl) 缓存
├── reports/        # 执行生成的 HTML/JSON 报告
├── logs/           # 运行日志
└── tmp/            # 临时过程产物
```

---

## 5. 关键参数说明
所有 `scripts/*.sh` 脚本均支持以下通用参数：

| 参数 | 说明 | 对应环境变量 |
| :--- | :--- | :--- |
| `--host` | 远程设备 IP | `REMOTE_HOST` |
| `--user` | SSH 用户名 | `REMOTE_USER` |
| `--password` | SSH 密码 | `REMOTE_PASS` |
| `--remote-dir` | 远程部署根路径 | `REMOTE_DIR` |
| `--output-dir` | (仅回传脚本) 本地保存路径 | - |

## 6. 注意事项与 FAQ

1.  **关于 psutil 安装失败**：
    *   `psutil` 需要远程环境有 `gcc` 和 `Python.h` 才能完成编译安装。
    *   **现象**：安装失败会抛出 `WARN`，但**不会中断**部署流程。
    *   **后果**：仅会导致 Dashboard 无法展示 CPU/内存监控指标，不影响测试用例逻辑执行。

2.  **测试失败判定**：
    *   例如 `eth_case` 或 `uart_case` 失败，通常是由于未插网线或未接回环工装导致的。这属于**有效测试结果**，系统会正常记录在报告中。

3.  **如何新增测试集？**
    *   在本地修改 `fixtures/xxx.json` 后，使用 `package_and_deploy_offline.py --fast-reuse` 快速同步即可。
