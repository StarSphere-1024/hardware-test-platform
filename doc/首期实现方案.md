### 0.1 文档目的

本文档用于定义首期实现范围、落地顺序、交付标准和阶段性取舍。本文档不替代软件设计文档，而是作为其首期执行版本。

---

## 1. 首期目标

首期实现目标是跑通单机 Linux 平台上的最小可用测试内核，覆盖以下能力：

1. 读取并解析全局配置、board profile、case、fixture。
2. 生成 ResolvedExecutionConfig。
3. 构建 fixture -> case -> function 任务图。
4. 执行顺序调度、超时控制、重试控制和 stop_on_failure。
5. 记录统一结果快照与事件流。
6. 跑通至少一个 fixture 的端到端执行。

---

## 2. 首期范围

### 2.1 In Scope

- Linux 本地平台适配
- 配置加载、校验、解析
- 领域模型与执行模型落地
- Scheduler 与 Runner 主链路
- Result Store / Event Store 最小实现
- 基于快照与事件流的终端 Dashboard 最小展示
- CLI 主入口：run_function、run_case、run_fixture
- 首批高频模块验证：ETH、UART、RTC、GPIO、I2C

### 2.2 Out of Scope

- Web Dashboard
- 云端上传
- 完整远程执行
- Zephyr/MCU 真正落地实现
- 全量 legacy 资产迁移
- 分布式调度与复杂资源编排

## 4. 实现顺序

### 4.1 第一阶段：配置链路

目标：生成可执行的 ResolvedExecutionConfig。

工作项：

1. 定义配置模型。
2. 实现配置加载器。
3. 实现配置校验器。
4. 实现优先级合并和模板解析。
5. 输出配置来源追踪。

完成标准：

- 可以从 fixture 或 case 输入生成完整配置对象。
- 配置错误可以在执行前阻断。

### 4.2 第二阶段：执行链路

目标：跑通任务图和调度主链路。

工作项：

1. 定义 ExecutionPlan、ExecutionTask、ExecutionContext。
2. 实现 FixtureRunner 和 CaseRunner。
3. 实现 Scheduler。
4. 实现 FunctionExecutor。
5. 实现基本重试和超时策略。

完成标准：

- 可以顺序执行一个 fixture。
- 可以正确生成 case/function 结果。

### 4.3 第三阶段：平台能力

目标：提供 Linux 首套能力实现。

工作项：

1. 定义 adapter 最小抽象。
2. 实现 LinuxAdapter。
3. 建立 capability 注册机制。
4. 提供 network、serial、rtc、gpio、i2c 首批能力。

完成标准：

- 首批模块能够通过 capability 调用平台能力。
- Function 不直接依赖原始配置文件。

### 4.4 第四阶段：可观测性

目标：实现事件流、结果快照和最小报告。

工作项：

1. 实现 Result Store。
2. 实现 Event Store。
3. 实现最小报告生成器。
4. 将执行链路与事件、结果打通。

完成标准：

- 可以输出当前状态快照。
- 可以记录完整事件序列。
- 可以生成最小文本或 JSON 报告。

### 4.5 第五阶段：首批模块验证

目标：用真实模块验证模型成立。

工作项：

1. 接入 ETH。
2. 接入 UART。
3. 接入 RTC。
4. 接入 GPIO。
5. 接入 I2C。
6. 运行 quick_validation fixture。

完成标准：

- 至少一个 fixture 可稳定运行。
- 执行结果、快照、事件、报告一致。

### 4.6 按模块任务清单

以下任务清单用于指导首期文件和模块的实际落地顺序。原则是先打通基础链路，再补足平台能力，最后用真实模块验收设计。

#### 4.6.1 模块 A：配置模块

目标：产出唯一可信的 ResolvedExecutionConfig。

涉及文件：

- framework/config/models.py
- framework/config/loader.py
- framework/config/validator.py
- framework/config/resolver.py
- framework/config/errors.py

任务清单：

1. 定义 GlobalConfig、BoardProfile、CaseSpec、FixtureSpec、ResolvedExecutionConfig 模型。
2. 实现配置文件加载规则，包括相对路径和默认路径解析。
3. 实现 schema 校验和错误类型归类。
4. 实现优先级覆盖链：global -> board profile -> fixture -> case -> function defaults -> CLI override。
5. 实现模板解析和配置来源追踪。

完成标准：

- 能从 fixture 输入得到完整配置对象。
- 配置错误在执行前失败，并带字段路径信息。

#### 4.6.2 模块 B：领域模型模块

目标：建立稳定的数据契约，避免执行逻辑直接操作松散字典。

涉及文件：

- framework/domain/requests.py
- framework/domain/specs.py
- framework/domain/execution.py
- framework/domain/results.py
- framework/domain/events.py

任务清单：

1. 定义 ExecutionRequest。
2. 定义 ExecutionPlan、ExecutionTask、ExecutionContext。
3. 定义 ExecutionResult、ResultSnapshot、ReportArtifact。
4. 定义 ExecutionEvent、EventRecord 和状态枚举。
5. 明确各对象的序列化格式。

完成标准：

- 调度层和观测层只依赖领域对象，不直接依赖原始 dict。
- 所有关键对象可序列化为 JSON 兼容结构。

#### 4.6.3 模块 C：执行模块

目标：跑通 fixture -> case -> function 的统一执行主链路。

涉及文件：

- framework/execution/scheduler.py
- framework/execution/fixture_runner.py
- framework/execution/case_runner.py
- framework/execution/function_executor.py
- framework/execution/policies.py
- framework/execution/errors.py

任务清单：

1. 实现 FixtureRunner，将 fixture 转换为根任务图。
2. 实现 CaseRunner，将 case 转换为 function 子任务并完成预检入口。
3. 实现 Scheduler 的 sequential 执行能力。
4. 实现 timeout、retry、stop_on_failure 的基础策略。
5. 实现 function、case、fixture 三级结果聚合。

完成标准：

- 至少可顺序执行一个 fixture。
- 失败、超时、重试行为能够正确反映到结果模型中。

#### 4.6.4 模块 D：平台适配模块

目标：提供 Linux 首套平台能力，实现首期模块调用所需的抽象。

涉及文件：

- framework/platform/registry.py
- framework/platform/adapters/base.py
- framework/platform/adapters/linux.py
- framework/platform/capabilities/network.py
- framework/platform/capabilities/serial.py
- framework/platform/capabilities/gpio.py
- framework/platform/capabilities/i2c.py
- framework/platform/capabilities/rtc.py
- framework/platform/capabilities/system_info.py

任务清单：

1. 定义 adapter 最小抽象和 capability 注册机制。
2. 实现 LinuxAdapter 的基础执行能力。
3. 实现 network capability。
4. 实现 serial capability。
5. 实现 rtc、gpio、i2c 首批 capability。

完成标准：

- ETH、UART、RTC、GPIO、I2C 五类模块可通过 capability 访问平台。
- Function 不直接读取 board profile 或调用散乱 shell 逻辑。

#### 4.6.5 模块 E：可观测性模块

目标：让执行状态、事件和报告具备统一事实来源。

涉及文件：

- framework/observability/result_store.py
- framework/observability/event_store.py
- framework/observability/report_generator.py
- framework/observability/logger.py

任务清单：

1. 实现 Result Store 原子写入当前快照。
2. 实现 Event Store 顺序写入执行事件。
3. 实现最小报告生成器。
4. 实现统一日志入口。
5. 将执行模块与事件、结果、日志打通。

完成标准：

- 执行过程中可看到完整事件链路。
- 执行结束后可获得可用报告与快照。

#### 4.6.6 模块 F：CLI 模块

目标：提供首期统一用户入口。

涉及文件：

- framework/cli/common.py
- framework/cli/run_fixture.py
- framework/cli/run_case.py
- framework/cli/run_function.py

任务清单：

1. 定义通用 CLI 参数和错误输出格式。
2. 实现 run_fixture 主入口。
3. 实现 run_case 主入口。
4. 实现 run_function 主入口。
5. 将 CLI 参数映射为 ExecutionRequest。

完成标准：

- 用户可通过单一入口触发三类执行。
- CLI 不直接承载业务调度逻辑。

#### 4.6.7 模块 G：首批函数与配置资产

目标：用高频模块验证新内核设计成立。

涉及目录：

- functions/network/
- functions/uart/
- functions/rtc/
- functions/gpio/
- functions/i2c/
- cases/
- fixtures/
- config/boards/

任务清单：

1. 实现 test_eth_ping 或同等最小网络函数。
2. 实现 test_uart_loopback 或同等最小串口函数。
3. 实现 test_rtc_read_write。
4. 实现 test_gpio_get_set。
5. 实现 test_i2c_scan 或 EEPROM 最小读写函数。
6. 编写对应 case JSON。
7. 编写 quick_validation fixture。

完成标准：

- 五类高频模块至少各有一个 function 可运行。
- quick_validation fixture 可稳定通过。

#### 4.6.8 模块 H：测试模块

目标：确保主链路具备基础回归能力。

涉及目录：

- tests/config/
- tests/domain/
- tests/execution/
- tests/platform/
- tests/observability/
- tests/smoke/

任务清单：

1. 为配置解析器补单元测试。
2. 为领域对象序列化补单元测试。
3. 为 Scheduler 顺序执行和重试补单元测试。
4. 为 LinuxAdapter 和首批 capability 补最小测试。
5. 编写 quick_validation 冒烟测试。

完成标准：

- 配置、执行、平台、观测四条链路都有基础测试覆盖。
- 冒烟测试能覆盖至少一个 fixture 的主流程。

### 4.7 推荐落地顺序

从依赖关系看，建议实际开发顺序为：

1. 模块 A：配置模块
2. 模块 B：领域模型模块
3. 模块 C：执行模块
4. 模块 D：平台适配模块
5. 模块 E：可观测性模块
6. 模块 F：CLI 模块
7. 模块 G：首批函数与配置资产
8. 模块 H：测试模块

这个顺序的原因是：

1. A 和 B 提供所有后续模块的基础输入和契约。
2. C 决定主执行链路是否成立。
3. D 和 E 分别补齐“怎么执行”和“怎么观测”。
4. F 只是用户入口，应在主链路稳定后接入。
5. G 和 H 用于验证而不是定义内核。

## 5. 首期阶段性取舍

### 5.1 明确接受的简化

1. 首期只支持 Linux 本地平台。
2. 首期仅实现最小 capability 集合。
3. 首期不要求 Web Dashboard，但允许提供基于快照与事件流的终端 Dashboard 最小展示。
4. 首期资源锁可以先不做独立复杂实现，但要保留设计扩展点。
5. 首期报告格式可以先 JSON 优先，再补文本格式。

### 5.2 明确不能退让的点

1. 配置必须先解析后执行。
2. Scheduler 必须是唯一调度核心。
3. Function 不得直接读取 board profile。
4. 事件和结果模型必须先立住。
5. 目录结构不能因为迁移期需求污染内核边界。

### 5.3 Dashboard 实现补充说明

为满足首期调试和回归验证需要，当前实现允许在不引入 Web 前端的前提下提供终端 Dashboard。

补充约束如下：

1. 当前落地的是 CLI Dashboard，不是 Web Dashboard。
2. Dashboard 必须直接消费当前框架的统一事实源，包括：
  - tmp/*_snapshot.json
  - logs/events/*.json
  - logs/*.log
  - reports/*
3. Dashboard 不再依赖 legacy 的 *_result.json 状态文件。
4. 系统监控信息允许通过独立 monitoring 模块输出到 tmp/system_monitor.json，作为增强信息而不是主执行事实源。
5. Dashboard 入口与当前 CLI 风格保持一致，支持独立运行，不得把展示逻辑耦合回 Scheduler。

## 6. 首期验收标准

1. 可以通过 CLI 运行一个 fixture。
2. 可以生成 ResolvedExecutionConfig 快照。
3. 可以输出 function、case、fixture 三级结果。
4. 可以记录事件流并按顺序重建执行时间线。
5. 至少 5 个高频模块通过新链路验证。
6. 新增一个 function 时不需要修改 Scheduler。

---

## 7. 风险与缓解

### 7.1 配置设计过度

风险：首期在配置层花费过多时间，迟迟无法跑通主链路。

缓解：先实现最小字段集，再补扩展字段。

### 7.2 平台抽象过早泛化

风险：为未来平台设计过多抽象，拖慢 Linux 首期落地。

缓解：先写 Linux 首套实现，再反推抽象边界。

### 7.3 首批模块无法验证模型

风险：模型设计看起来合理，但一落到真实模块就需要大量返工。

缓解：优先用 ETH、UART、RTC、GPIO、I2C 做早期验证。
